⚡ What is Energy Meter Tracker?
⚠️ Informational use only. Energy Meter Tracker provides a close approximation of your consumption based on local sensor readings. It cannot replicate your supplier's authoritative meter reconciliation, which is derived from your smart meter's secure data stream and subject to settlement adjustments. Do not use this data for billing disputes or formal energy accounting.

Required Hardware The addon is intended to be used with an IHD/CAD capable of providing meter readings at reliable regular intervals (0.1Hz).

Energy Meter Tracker is a Home Assistant add-on that records your electricity usage in configurable reconciliation periods — matching the intervals used by your energy supplier for billing. The add-on stores meter readings at each boundary so you can see what happened in any given period, including how much of your usage was from the grid versus your battery, EV or other device.

It works by subscribing to your smart meter sensors in Home Assistant and taking timestamped readings continuously. At each reconciliation period boundary it interpolates precisely to the boundary timestamp, finalises the block, and moves on — giving you accurate data even when sensor updates don't land exactly on the boundary.

🕐 Meter Reconciliation Period

Each block covers exactly one reconciliation period: for example 09:00 → 09:30 at 30 minutes, or 09:00 → 09:05 at 5 minutes. In the UK, energy suppliers use 30-minute half-hourly (HH) settlement periods. Other countries may use different resolutions — some markets are moving to 15-minute or even 5-minute intervals. Configure the reconciliation period to match your supplier before data collection begins. It cannot be changed without clearing all historical data.

Within each block the engine collects multiple readings from your sensors. When the block closes at the boundary, it uses the readings either side of the boundary timestamp to interpolate a precise opening and closing value — so even if your smart meter only updates every 60 seconds, the block delta is accurate to the boundary, not to whenever the last reading happened to arrive.

Why interpolation matters: If your meter reads 1000.0 kWh at 09:28 and 1000.5 kWh at 09:32, the engine calculates that approximately 0.25 kWh crossed the 09:30 boundary — not 0.0 kWh (last reading before) or 0.5 kWh (total since last reading).
⚖️ Main Meter & Sub-meters

The main meter reads directly from your smart meter via a Consumer Access Device (CAD) connected to your local MQTT broker, typically updating every 10 seconds. It is the authoritative source of truth for how much energy crossed the grid boundary. Everything else is derived from it.

Sub-meters track individual devices — an EV charger, home battery, or heat pump. They tell you how that grid energy was used, but they do not override the main meter reading. Solar generation is not currently supported as a sub-meter type.

Grid remainder
In each block, sub-meter consumption is subtracted from the main meter's grid import. Whatever is left is the remainder — the portion of grid import not accounted for by any sub-meter. This is what's used for billing calculations, not the raw main meter total.
Battery / inverter logic
Devices marked inverter possible may show consumption from battery discharge rather than grid. The engine allocates grid energy to protected loads first (EV, heat pump), then to inverter devices. Any excess sub-meter kWh is attributed to battery discharge.
⚠️ Export sub-metering and solar generation are not yet supported. The export channel on the main meter records total grid export. Individual device export (e.g. battery discharging to grid, or V2G from a V2X-capable EV) is not yet broken down by sub-meter. Solar generation tracking is planned for a future release.
📡 Sensor Requirements

Main meter — Read sensors

  • Must be cumulative kWh — ever-increasing total, never resets
  • High-frequency updates preferred (10–60s) for better boundary interpolation
  • Typically from your CAD via MQTT integration
  • Example: sensor.smart_meter_electricity_import

Rate sensors

  • Must be the current rate per kWh — the live tariff rate
  • Used to calculate cost per block
  • For Agile/flexible tariffs these change each reconciliation period — the engine captures the rate in effect during each block
  • Example: sensor.octopus_energy_electricity_current_rate

Sub-meter — Read sensors

  • Must be cumulative kWh consumed (import only) — not net usage, not export, not power in watts
  • Can update less frequently (60s+) — interpolation still works
  • CT clamp via PowerCalc, or native device integration (Zappi, Solax, etc.)
  • If no rate sensor is configured the main meter import rate is used automatically

Standing charge sensor

  • Must be a daily standing charge value
  • Can be a fixed input_number helper if your supplier doesn't expose it
  • Only needed on the main import channel
🔧 Gap Filling

If the add-on is stopped or Home Assistant restarts, the engine detects any missing blocks when it comes back online. It uses the last known reading before the outage and the first reading after to interpolate the missing blocks, marking them as interpolated: true in the data.

Interpolated blocks are less accurate than live-captured blocks but ensure your cumulative totals remain consistent. Gaps longer than 12 hours are not interpolated — the blocks are left as zero to avoid misleading data.

💾 Data Storage & Backup

All data is stored in a single SQLite database:

  • blocks.db — everything: all finalised blocks, config history, meter definitions, and current block state. This is the only file you need to back up.
  • meters_config.json — human-readable config export, written automatically for reference. The database is authoritative — this file is not read back as live state.

Data is stored in the add-on's private /data/energy_meter_tracker/ directory. After every block finalise, blocks.db is copied to /share/energy_meter_tracker_backup/ using SQLite's online backup API — no risk of a corrupted mid-write copy. A zip snapshot is also created automatically before every config save, and manually on demand from the Data Management page.

⚠️ Uninstalling the add-on will wipe /data/. Always ensure you have a recent backup in /share/ before uninstalling. Updating the add-on (without uninstalling) preserves your data.
🐳 Standalone Docker

When running in standalone Docker mode, the add-on connects to your HA instance using a Long-Lived Access Token rather than the Supervisor. The UI is accessible directly at http://<host>:8099.

Adding to the HA sidebar — add a panel_iframe entry to your configuration.yaml: 

panel_iframe:
  energy_meter:
    title: "Energy Meter"
    icon: mdi:speedometer
    url: "http://192.168.1.x:8099"

Replace 192.168.1.x with your Docker host IP and restart HA. This adds a sidebar entry that embeds the UI within HA — similar to the supervised experience.

⚠️ The volume mount is essential. Always run with -v /path/to/data:/data/energy_meter_tracker — without it all data is lost when the container is recreated. The /share backup path is not available in standalone mode so use the Backup Now button regularly.
🏠 Home Assistant Sensors

After each block finalises, the engine updates four synthetic HA sensors with your cumulative totals:

  • sensor.energy_meter_import_kwh — total grid import (kWh)
  • sensor.energy_meter_export_kwh — total grid export (kWh)
  • sensor.energy_meter_import_cost — total import cost (£)
  • sensor.energy_meter_export_credit — total export credit (£)

These sensors use state_class: total_increasing and device_class: energy so they are compatible with the HA Energy dashboard and Utility Meter integrations. You can point Utility Meter helpers at these sensors to get daily, weekly, and monthly breakdowns.
